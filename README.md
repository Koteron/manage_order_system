# Order Management System

Данный проект представляет собой микросервис на базе FastAPI, реализующий отказоустойчивую архитектуру управления заказами. Основной акцент сделан на обеспечении консистентности данных в распределенной среде и надежной доставке сообщений.
Инструкции по установке и запуску
Проект полностью контейнеризирован. Все необходимые переменные окружения предварительно настроены в файле .env для упрощения демонстрационного запуска.

## Запуск проекта

#### Предварительные требования
Убедитесь, что в системе установлен Docker и Docker Compose.

```bash
docker compose up --build
```
    Для удобства проверяющего .env было залито в репозиторий. Обычно так делать не стоит

- Swagger UI: http://localhost:8000/docs
- ReDoc: http://localhost:8000/redoc

## Техническое задание: Разработка сервиса управления заказами

### 1. Введение
Разработать сервис управления заказами на **FastAPI**, поддерживающий аутентификацию, работу с очередями сообщений, кеширование и фоновую обработку задач.

### 2. Функциональные требования

#### 2.1 API эндпоинты

| Метод | URL | Описание |
| :--- | :--- | :--- |
| **POST** | `/register/` | Регистрация пользователя (email, пароль) |
| **POST** | `/token/` | Получение JWT-токена (OAuth2) |
| **POST** | `/orders/` | Создание заказа (Только авторизованные) |
| **GET** | `/orders/{order_id}/` | Получение заказа (сначала из Redis) |
| **PATCH** | `/orders/{order_id}/` | Обновление статуса заказа |
| **GET** | `/orders/user/{user_id}/` | Получение заказов пользователя |

#### 2.2 База данных (PostgreSQL)
**Таблица `orders`:**
*   `id` (UUID, primary key)
*   `user_id` (int, ForeignKey на пользователей)
*   `items` (JSON, список товаров)
*   `total_price` (float)
*   `status` (enum: `PENDING`, `PAID`, `SHIPPED`, `CANCELED`)
*   `created_at` (datetime)

#### 2.3 Очереди сообщений (Kafka / RabbitMQ)
*   RabbitMQ/Kafka используется как брокер сообщений между сервисами (**event-bus**), а не как брокер Celery.
*   При создании заказа сервис публикует событие `new_order` в очередь.
*   Отдельный consumer подписывается на очередь, получает сообщения `new_order`, выполняет обработку и запускает фоновую задачу в Celery/taskiq.
*   Celery/taskiq используется только для выполнения фоновых задач и не читает RabbitMQ/Kafka напрямую.

#### 2.4 Redis (Кеширование заказов)
*   Если заказ запрашивается повторно — отдавать его из кеша (**TTL = 5 минут**).
*   При изменении заказа — обновлять кеш.

#### 2.5 Celery/taskiq (Фоновая обработка)
*   Фоновая задача обработки заказа: `time.sleep(2)` и `print(f"Order {order_id} processed")`.

#### 2.6 Безопасность
*   JWT-аутентификация (OAuth2 Password Flow).
*   CORS-защита (ограничение кросс-доменных запросов).
*   Rate limiting (ограничение частоты запросов на API).
*   SQL-инъекции — использование только ORM-запросов.

### 3. Нефункциональные требования
*   Использование **FastAPI** с **Pydantic**.
*   Работа с PostgreSQL через **SQLAlchemy + Alembic**.
*   Асинхронное взаимодействие с Kafka / RabbitMQ.
*   **Docker Compose** для развертывания всей инфраструктуры.

## Описание реализации

### Архитектурные паттерны

В системе реализовано сочетание нескольких инфраструктурных решений для обеспечения надежности:

- `Transactional Outbox`<br>
Для исключения рассинхронизации между базой данных и брокером сообщений Kafka применен паттерн Outbox.

    Данные о заказе и соответствующее событие сохраняются в PostgreSQL в рамках единой ACID-транзакции.
    Фоновый воркер (Outbox Publisher) опрашивает таблицу событий и пересылает их в Kafka, используя транзакционный режим (Exactly-Once Semantics). Это гарантирует, что сообщение будет отправлено хотя бы один раз и не будет потеряно при сбое сети.

- `Идемпотентность и Deduplication (Redis)`<br>
Чтобы избежать повторной обработки одного и того же события (например, при дублировании сообщений в Kafka), Consumer использует Redis:

    Перед обработкой проверяется наличие ключа события в Redis с использованием атомарной операции SET NX.
    Если событие уже обрабатывалось, оно игнорируется. Это обеспечивает безопасную работу в распределенной среде.

- `Dual-Broker (Kafka + RabbitMQ)`<br>
Проект разделяет зоны ответственности между брокерами:

    - `Apache Kafka`<br> 
    Выступает в роли Event Bus для межсервисного взаимодействия и хранения истории событий.
    - `RabbitMQ`<br> 
    Используется как брокер задач для Celery.

- `Безопасность`<br>

    - `JWT + Bcrypt`<br> 
    Реализована полноценная авторизация. Пароли хранятся в виде хешей (bcrypt), доступ к эндпоинтам защищен токенами.
    - `Rate Limiting`<br> 
    Использование SlowAPI для защиты API от перегрузок и перебора паролей (10 запросов в минуту на чувствительные эндпоинты).

## Технологический стек

    FastAPI: Основной веб-фреймворк.
    PostgreSQL + SQLAlchemy 2.0 (Async): Хранение данных.
    Alembic: Верификация схемы БД при запуске.
    Apache Kafka (KRaft): Стриминг событий.
    RabbitMQ + Celery: Фоновые задачи.
    Redis: Кэширование и идемпотентность.
